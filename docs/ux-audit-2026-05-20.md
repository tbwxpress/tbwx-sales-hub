# Sales Hub UX Audit — 2026-05-20

**Auditor:** Claude (Opus 4.7)
**Scope:** Telecaller workflow, agent stats accuracy, pipeline stage management, lead detail UX
**Personas tested:** (a) Telecaller assigned via auto-queue, (b) Owner agent reviewing telecaller work, (c) Admin viewing agent stats dashboard
**Method:** Code-level audit of permissions, API gating, UI behaviour, and activity-tracking flow (no live browser session — codebase is the source of truth).

---

## TL;DR — The Three Things to Fix First

1. **Notes API silently drops `created_by`** — every note in the system is stored with an empty author string. The Daily Activity tracker filters out empty `created_by`, so 100% of notes are invisible in stats. Single-line UI fix.
2. **Telecallers can see the status dropdown but the API rejects them** — UI shows it (rendered unconditionally on line 1099 of the lead detail page), backend returns 403 because the `lead.assigned_to !== user.name` check at line 53-60 of `api/leads/[id]/route.ts` doesn't account for `lead_telecaller_assignments`. Confusing UX + no path forward.
3. **Telecaller activity dashboard counts the wrong things** — current "Calls (7d) / Notes (7d)" columns are accurate but limited. There's no count of "messages sent" (telecallers can send messages and they're tracked), no count of "stage-change requests" (doesn't exist yet), and no "ghost telecaller" warning equivalent to the closer warning.

Fix these three and the user's reported "telecaller activity always 0" disappears immediately.

## Detailed Findings

### F1 [Critical] — Every note loses its author

**Where:**
- `src/app/leads/[id]/page.tsx:76-80` — `LeadNotes.handleSave` posts `{ note: newNote.trim() }` only
- `src/app/api/inbox/[phone]/notes/route.ts:22, 28` — accepts `created_by` from request body, falls back to `''`
- `src/app/api/analytics/agent-activity/route.ts:92` — `WHERE created_at >= ? AND created_at < ? AND created_by != ''` filters out empty authors

**Impact:** All notes from all users (closers, telecallers, admin) are stored with `created_by = ''` and never counted in Daily Activity. The lead detail page also displays `n.created_by` (line 114) which renders blank for all notes.

**Fix:** Pull `user.name` from session inside the route — don't accept it from the body. This is also a small security improvement (clients can't spoof an author).

```ts
// notes/route.ts POST handler
const session = await getSession()
const user = requireAuth(session)
// ...
const id = await insertNote({ phone, note: note.trim(), created_by: user.name })
```

This is 6 lines of code. It fixes notes attribution for the entire history of notes going forward and unblocks all telecaller note activity in the stats dashboard.

### F2 [Critical] — Telecaller sees status dropdown, gets silent 403

**Where:**
- `src/app/leads/[id]/page.tsx:1099-1108` — `<select value={lead.lead_status} onChange={(e) => updateField('lead_status', e.target.value)}>` rendered unconditionally
- `src/app/api/leads/[id]/route.ts:53-60` — gate `if (lead?.assigned_to !== user.name && !(can_assign && unassigned)) → 403`

**Telecaller perspective:**
1. Open assigned lead → status dropdown is visible and enabled
2. Click "CALL_DONE_INTERESTED" → toast says "Status updated" silently fails (the `updateField` swallows the error — line 625 `} catch { /* silent */ }`)
3. Refresh page → status is back to what it was
4. Confusion: "did I just lose the change?"

**Fix path A (safest, immediate):** Disable the dropdown for telecallers with a hint:

```tsx
const canEditStatus = sessionUser?.role === 'admin' ||
                      (lead.assigned_to === sessionUser?.name)
// Render disabled + "Propose change ↓" button if !canEditStatus
```

**Fix path B (Gavish's actual ask):** Replace the dropdown with a "Propose status change" workflow — see F5.

I recommend path B because it gives telecallers an actionable next step instead of just a wall, and it gives the owner agent a review queue.

### F3 [High] — Telecallers blocked from all lead field updates

Same gate as F2 — line 53-60 of `api/leads/[id]/route.ts`. A telecaller assigned through `lead_telecaller_assignments` cannot:
- Set `next_followup` (their main job after a "Callback Scheduled" outcome!)
- Set `attempted_contact` flag
- Update `lead_priority`
- Set `first_call_date`

**Fix:** The PATCH route needs to also allow updates when the user is the assigned telecaller via `lead_telecaller_assignments`. Pseudo:

```ts
if (user.role === 'agent') {
  const isOwner = lead?.assigned_to === user.name
  const isUnassigned = !lead?.assigned_to
  const isAssignedTelecaller = await isLeadTelecaller(rowNum, user.id)

  const allowedAsTelecaller = ['next_followup', 'attempted_contact', 'first_call_date', 'lead_priority']
  const onlyTelecallerFields = isAssignedTelecaller &&
    Object.keys(body).every(k => allowedAsTelecaller.includes(k))

  if (!isOwner && !(user.can_assign && isUnassigned) && !onlyTelecallerFields) {
    return 403
  }
}
```

Status changes from telecallers go through the new proposal endpoint (F5), not PATCH.

### F4 [High] — Idle telecallers don't trigger a ghost-agent warning

**Where:** `src/app/api/analytics/agent-activity/route.ts:266-271` only fires the warning when `a.type === 'closer'`. Telecallers with auto-queue assignments but zero touches get a green-bordered card with `0 leads touched` and no alert.

**Fix:** Extend the ghost detection to telecallers. A telecaller with `total_queue >= N` (e.g. 5) and `leads_touched === 0` for the day is a ghost. The warning text should differ — "X was queued Y leads but didn't touch one" rather than "assigned Y new leads".

The telecaller's queue size for the day = manual_assigned + auto_queue eligible at `dateStr`. We already have those numbers in the telecaller stats route — they just need to flow into the activity route.

### F5 [High] — No "propose stage change" workflow

This is the centrepiece of what Gavish asked for. Current state:
- Telecaller can talk to the lead, take a call, add a note
- Telecaller CANNOT move the lead from NO_RESPONSE → CALL_DONE_INTERESTED → HOT
- Owner agent has no signal that the telecaller has made progress unless they happen to read the notes

**Proposed design:**

**New table** `lead_stage_proposals`:
```sql
CREATE TABLE lead_stage_proposals (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  lead_row INTEGER NOT NULL,
  phone TEXT NOT NULL,
  proposed_by_user_id TEXT NOT NULL,
  proposed_by_name TEXT NOT NULL,
  current_status TEXT NOT NULL,
  proposed_status TEXT NOT NULL,
  reason TEXT NOT NULL,
  status TEXT NOT NULL DEFAULT 'pending' CHECK(status IN ('pending', 'approved', 'rejected', 'auto_expired')),
  reviewed_by_user_id TEXT,
  reviewed_by_name TEXT,
  reviewed_at TEXT,
  review_notes TEXT,
  created_at TEXT DEFAULT (datetime('now')),
  UNIQUE(lead_row, status) WHERE status = 'pending'  -- one pending per lead
);
CREATE INDEX idx_stage_proposals_lead ON lead_stage_proposals(lead_row);
CREATE INDEX idx_stage_proposals_status ON lead_stage_proposals(status);
```

**New endpoints:**
- `POST /api/leads/[id]/stage-proposal` — telecaller creates proposal (requires assignment row or telecaller flag)
- `GET /api/stage-proposals?owner=<name>&status=pending` — owner/admin lists pending
- `POST /api/stage-proposals/[id]/approve` — owner/admin approves → triggers PATCH internally
- `POST /api/stage-proposals/[id]/reject` — owner/admin rejects with optional `review_notes`

**UI flow:**
- Lead detail page (telecaller view): status dropdown replaced with `<button>Propose status change</button>`. Click → modal with proposed_status select + reason textarea. Submit → notification fires to lead owner.
- Lead detail page (owner view): if a pending proposal exists, an amber banner appears at top: `"Telecaller {name} proposes: NO_RESPONSE → CALL_DONE_INTERESTED — 'Spoke to him, wants outlet visit Saturday.' [Approve] [Reject]"`. Approving applies the status change as if the owner did it manually.
- Owner dashboard: badge `"3 proposals awaiting review"` linking to `/proposals`.
- Admin dashboard (agent-stats): new column `Proposals (submitted / approved-rate)` per telecaller, e.g. `12 / 75%`.

**Why this works:**
- Telecaller has a real action with an audit trail — counts in stats, visible to manager
- Owner agent retains final say — quality bar holds
- Approve-rate becomes a leading indicator of telecaller calibration (low approve-rate = telecaller is being sloppy or aggressive)
- Reason field forces the telecaller to justify, which is exactly the "make sure they're doing their job well" check Gavish asked for

**Auto-expiry:** Proposals older than 72 hours with no review move to `auto_expired`. Otherwise pending queue grows forever.

### F6 [High] — Lead detail page has no telecaller view

Currently the page renders the same UI for everyone except the small `canEdit` gate on the Lead Info card (line 795). Telecallers see:
- Assigned To dropdown (they can't change it, but it's there)
- Agreement generator button (they can't fill it, but it's there)
- All the closer-only sidebar cards

**Fix:** Add a `viewMode` derived from the session:
- `'closer'` (default, current UI)
- `'telecaller'` — hides Manage card, Agreements, Voice Agent settings; promotes Notes + Call Log + Quick Replies + Propose Status panel

The telecaller layout becomes a focused tool: lead info (read-only), conversation, call log, notes, status proposal. Three less columns of visual noise, one click closer to the things they actually do.

### F7 [Medium] — `/today` is a feed, not a queue

`/today` mixes hot-stale, unread replies, telecaller handoffs, overdue follow-ups, and new assignments. For a closer this is the right home; for a telecaller their job is much narrower: "call the next person, log the outcome, move on."

**Proposed addition:** `/queue` or `/today/calls` for users with `is_telecaller`. Renders a single ranked list of leads in their queue, sorted by:
1. Followup overdue (descending overdue days)
2. Status priority (HOT > REPLIED > CALL_DONE_INTERESTED > everything else)
3. Lead age (newer first within the same status)

Each row has a single big `[Call now →]` button that opens the lead detail with the Log Call modal pre-armed.

This collapses a telecaller's daily flow to: open queue → click row 1 → call → modal → outcome → modal closes → row 1 is gone → click new row 1. No navigation between dashboard sections.

### F8 [Medium] — Session JWT lacks closer/lead-pool flags

`src/lib/auth.ts:46-55` puts `is_telecaller` in session but not `is_closer` or `in_lead_pool`. Several endpoints (analytics, leads list) re-fetch the user row to read these flags. Minor performance cost, more importantly it's a source of "stale session" bugs.

**Fix:** Add `is_closer` and `in_lead_pool` to the session payload. Bumping the cookie name (or invalidating via `JWT_SECRET` rotation) forces re-login for all users.

This is low urgency — current re-fetch works correctly — but worth doing alongside the bigger telecaller work since both areas of code touch the same user object.

---

### F9 [Medium] — LogCallModal is 4 fields for every call

For "no answer" — the most common outcome at the start of an outreach round — the telecaller has to: open modal → leave duration blank → pick "No Answer" from dropdown → leave notes blank → click "Save Call Log". 5 clicks/interactions for an empty event.

**Fix:** Add an outcome-row of buttons above the form:

```
[ 📞 No answer ]  [ 📵 Busy ]  [ ❌ Wrong number ]  [ ✅ Picked up → fill below ]
```

The first three log the call instantly and close the modal. The fourth expands the form. Reduces "no answer" logging from 5 interactions to 1, and that's the most common outcome.

---

### F10 [Medium] — Status changes have no "why"

`lead_status_changes` table captures `old_status, new_status, changed_by, source` but not a reason. When auditing a lead later there's no way to know "why did Pankaj move this from HOT to LOST on 5/12?" Combined with F5 (proposal reason), the system could finally have a complete history.

**Fix:** Add `reason TEXT` to `lead_status_changes`. When an owner approves a proposal, the proposal's `reason` field flows in. When status is changed directly via PATCH, optionally accept a `status_change_reason` field on the body — UI prompts only on status moves that matter (HOT, CONVERTED, LOST).

---

### F11 [Low] — Frontend recomputes server-grouped data

`src/app/agent-stats/page.tsx:101-186` recomputes per-agent metrics by iterating all leads in the browser, even though `/api/analytics/agent-activity` already does grouping server-side. The pipeline view (`/pipeline`) does similar work.

**Fix:** Move `calculateAgentMetrics` into the analytics route or a new `/api/analytics/agent-totals` endpoint. Cleaner test surface and faster page load on large lead lists.

Low urgency, schedule alongside other refactors.

### F13 [High] — Agents and telecallers can't see their own stats

**Where:**
- `src/app/agent-stats/page.tsx:234-237` — non-admins are redirected to `/dashboard`
- `src/app/api/analytics/agent-activity/route.ts:50` — `requireAdmin(user)` blocks the API entirely

**Impact:** A closer who's having a great day doesn't see it. A telecaller who completes 20 calls has no feedback. The only person who knows their numbers is Gavish, who has to tell them. Removes a free motivation lever and forces a 1:1 channel for what should be ambient.

**Privacy consideration:** Showing peer-by-peer leaderboards in a small team can backfire (anxiety, gaming the metrics). Common SaaS pattern: **your numbers + team average + your rank, never named peer-by-peer**.

**Proposed design:**

Two views from the same endpoint:

1. **Admin view (current):** full leaderboard, all agents, ghost warnings, telecaller table.
2. **Self view (new):** your own card (same metrics as admin sees for you) + a "team average" row + your rank shown as `#3 of 7` without naming the other six.

API change — `/api/analytics/agent-activity` accepts non-admin sessions and returns:
```ts
{
  scope: 'self',
  date: '...',
  you: { /* same shape as one entry in `agents[]` for admin view */ },
  team_avg: { manual_messages, calls_logged, notes_added, status_changes, leads_touched },
  your_rank: { position: 3, of: 7 },  // ranked by leads_touched
}
```

UI: new `/my-stats` page (or repurpose `/agent-stats` to render the self view when `role === 'agent'`). Probably simpler to fork the route:
- `/agent-stats` → admin only (unchanged)
- `/my-stats` → all authenticated users, shows self view
- Add "My Stats" to the navbar for non-admins

**Bonus for telecallers:** their /my-stats also shows:
- Total queue size today
- Pending proposals they've submitted (once F5 ships)
- Approve-rate of past proposals
- Calls logged this week with sparkline

That last bullet is the actual motivator. A telecaller seeing "you logged 47 calls this week, your highest ever" is going to log more next week.

**Where it slots in:** wave 1 fixes the data (notes bug) so a self-view actually shows numbers. Wave 2 adds proposals which feed the telecaller's self-view. So F13 lands in **Wave 2** alongside F5. The endpoint extension is small (~30 lines), the new page reuses the existing card components from `/agent-stats`.

### F12 [Low] — Telecaller queue size not surfaced anywhere

A telecaller signing in has no quick-glance "you have 47 leads to call" indicator. They have to go to /dashboard and notice the filtered list is non-empty.

**Fix:** Navbar badge for `is_telecaller` users showing queue size. Number links to `/queue` (per F7).

## What I'd Build First If You Could Only Do One Thing

Wave 1 alone — 1 to 2 hours of work — solves the "telecaller stats are always 0" pain immediately, because **the notes bug (F1) accounts for the bulk of invisible work telecallers are already doing today**. They're adding notes; the notes are landing without an author; the analytics filter discards them. Single-character commit message: "fix(notes): capture author from session".

Wave 2 is the medium-term answer to "make agents review the telecaller's work" — without it, the telecaller is still constrained to a side channel.
